# Бюджет: расчёт «в день» с учётом обязательных расходов

Дата: 2026-07-21

## Проблема

На вкладке «Бюджет» строка `budget-days-row` показывает, сколько можно тратить в день:

```
daily = (totalLimit − totalSpent) / daysLeft
факт  = totalSpent / daysGone
```

В `totalLimit` и `totalSpent` сидят аренда, ЖКУ, подписки и прочие расходы, которыми пользователь не управляет в течение дня. Цифра «в день» из-за этого не отвечает на реальный вопрос: сколько я могу потратить сегодня на еду, такси и всё живое.

Отдельная беда — момент оплаты. Если 20-го числа аренда 40 000₽ ещё не списана, «в день» весь месяц завышена на её величину, а в день списания обваливается.

## Терминология

Флаг уже существует в данных: `expense.special ∈ {null, 'oneoff', 'recurring'}` (`index.html:1146`), сейчас используется только для фильтра графика «День за днём».

Новая терминология:

- **Обязательные** — расходы с заполненным `special`. Подтипы сохраняются: `recurring` = «↻ регулярное», `oneoff` = «• разовое».
- **Переменные** — всё остальное. Именно ими пользователь распоряжается ежедневно.

Собирательное «Особые» в UI заменяется на «Обязательные». Модель данных не меняется, миграции нет.

## Расчёт

Новая функция `_budgetFree(y, m)` рядом с `renderBudget()` в секции `═══ budget.js ═══`.

```
recSpent  = Σ обязательных ↻ этого месяца
oneSpent  = Σ обязательных • этого месяца
recPlan   = среднее ↻ за 3 прошлых месяца
reserve   = max(recSpent, recPlan) + oneSpent      // резерв на месяц
varSpent  = totalSpent − recSpent − oneSpent       // потрачено переменных
varLimit  = totalLimit − reserve                   // бюджет переменных
varLeft   = varLimit − varSpent
daily     = varLeft / daysLeft
факт      = varSpent / daysGone
прогноз   = totalSpent + max(0, reserve − recSpent − oneSpent) + факт × daysLeft
```

Ключевая идея — `reserve` берётся по прогнозу, пока обязательные не оплачены, и замещается фактом, как только фактическая сумма превысит прогноз. Аренда «занимает место» в бюджете с 1-го числа, поэтому цифра «в день» не прыгает в день списания.

Разовые (`oneoff`) не прогнозируются — только факт текущего месяца. Прогнозировать их бессмысленно: холодильник покупается один раз.

Глубина среднего для `recPlan` — 3 месяца, фиксированно, без переключателя. Регулярные расходы меняются редко.

### Границы

- `varLimit < 0` (резерв съел весь лимит) → строка красная, `daily = 0`.
- Обязательных нет ни в этом месяце, ни в трёх прошлых → `reserve = 0`, формула вырождается в текущую, вторая строка не рисуется.
- Все суммы фильтруются по `!_deleted`, как везде в проекте.

## UI

Строка `budget-days-row` (`index.html:1629-1657`) считает по переменным:

```
📅 11 дн. до конца месяца · ✓ 1 240₽/день · факт 980₽/день · прогноз 47 200₽
обязательные 52 000₽ · ↻ 40 000 резерв · • 12 000 факт
```

Вторая строка мельче и приглушённее первой. Для `recurring` подпись «резерв», пока `recSpent < recPlan`, и «факт» после. Все денежные значения через `fmtH()` — иначе приватный режим их не скроет.

## Что не входит

- Ручной override резерва в редакторе лимитов — по YAGNI. Авто-прогноз самокорректируется; вернуться к этому, если на практике будет мазать.
- Переключатель «всего / переменные» на строке — лишняя механика, одна честная цифра лучше двух.
- Изменения в графиках и аналитике — фича живёт только на вкладке «Бюджет».

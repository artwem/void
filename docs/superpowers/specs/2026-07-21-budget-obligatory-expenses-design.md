# Бюджет: расчёт «в день» с учётом особых расходов

Дата: 2026-07-21

## Проблема

На вкладке «Бюджет» строка `budget-days-row` показывает, сколько можно тратить в день:

```
daily = (totalLimit − totalSpent) / daysLeft
факт  = totalSpent / daysGone
```

В `totalLimit` и `totalSpent` сидят аренда, ЖКУ, подписки и прочие расходы, которыми пользователь не управляет в течение дня. Цифра «в день» из-за этого не отвечает на реальный вопрос: сколько я могу потратить сегодня на еду, такси и всё живое.

Отдельная беда — момент оплаты. Если 20-го числа аренда 40 000₽ ещё не списана, «в день» весь месяц завышена на её величину, а в день списания обваливается.

## Терминология

Флаг уже существует в данных: `expense.special ∈ {null, 'oneoff', 'recurring'}` (`index.html:1146`), сейчас используется только для фильтра графика «День за днём».

Подтипы схлопываются в один булев тег — **«Особое»**. Различие «регулярное / разовое» из интерфейса уходит: в модалках расхода и шаблона остаётся выбор «Обычный / Особое», бейдж в списках один.

- **Особые** — расходы с truthy `special`.
- **Переменные** — всё остальное. Именно ими пользователь распоряжается ежедневно.

Миграции данных нет. Новые записи пишут `special: true`, старые `'oneoff'`/`'recurring'` остаются в базе и читаются как truthy — весь существующий код проверяет `special` именно на truthy. При открытии старой записи на редактирование сегмент нормализует значение (`exp.special ? '1' : ''`), и при сохранении она станет `true`.

## Расчёт

Новая функция `_budgetFree(y, m, totalSpent, totalLimit)` рядом с `renderBudget()` в секции `═══ budget.js ═══`.

```
specSpent = Σ особых этого месяца
specPlan  = min(Σ особых за каждый из 3 прошлых месяцев)
reserve   = max(specSpent, specPlan)
varSpent  = totalSpent − specSpent
varLimit  = totalLimit − reserve
varLeft   = varLimit − varSpent
daily     = varLeft / daysLeft
факт      = varSpent / daysGone
прогноз   = totalSpent + max(0, specPlan − specSpent) + факт × daysLeft
```

Ключевая идея — `reserve` берётся по прогнозу, пока особые не оплачены, и замещается фактом, как только фактическая сумма превысит прогноз. Аренда «занимает место» в бюджете с 1-го числа, поэтому цифра «в день» не прыгает в день списания.

Прогноз — **минимум**, а не среднее, потому что тег больше не различает регулярное и разовое. Минимум за три месяца приближает ту часть особых расходов, которая повторяется каждый месяц (аренда, ЖКУ, подписки), и отсекает разовые всплески: холодильник за 80 000₽ попадёт только в один месяц из трёх и в минимум не пройдёт.

Месяцы без единого расхода пропускаются — иначе новый пользователь получит `specPlan = 0`. Если данных за прошлые месяцы нет вовсе, `specPlan = 0` и резерв работает по факту.

### Границы

- `varLimit < 0` (резерв съел весь лимит) → строка красная, `daily = 0`.
- Особых нет ни в этом месяце, ни в трёх прошлых → `reserve = 0`, формула вырождается в текущую, вторая строка не рисуется.
- Все суммы фильтруются по `!_deleted`, как везде в проекте.

## UI

Строка `budget-days-row` (`index.html:1629-1657`) считает по переменным:

```
📅 11 дн. до конца месяца · ✓ 1 240₽/день · факт 980₽/день · прогноз 47 200₽
особые 40 000₽ · резерв
```

Вторая строка мельче и приглушённее первой. Подпись «резерв», пока `specSpent < specPlan`, и «факт» после. Все денежные значения через `fmtH()` — иначе приватный режим их не скроет.

Переключатель «Особые | Без» над графиком «День за днём» и одноимённая секция в «Аналитике» не меняются — они и так оперируют собирательным понятием.

## Что не входит

- Ручной override резерва в редакторе лимитов — по YAGNI. Авто-прогноз самокорректируется; вернуться к этому, если на практике будет мазать.
- Переключатель «всего / переменные» на строке — лишняя механика, одна честная цифра лучше двух.
- Миграция старых `'oneoff'`/`'recurring'` в `true` — не нужна, truthy читается одинаково.

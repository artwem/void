# Особые расходы (special expenses)

**Дата:** 2026-06-19
**Проект:** V.O.I.D. (nto) — личные финансы PWA

## Проблема

Крупные расходы мешают восприятию «обычного дня»:
- **разовые** крупные покупки (новый ноут) — выбросы, портят средний день;
- **регулярные** крупные (аренда) — фон, заглушают мелкие траты.

Нужно помечать такие расходы и уметь не учитывать их в day-за-днём и в аналитике, при этом видеть их отдельно.

## Решение

Новое поле `special` на записи расхода с тремя состояниями. Day-таб показывает особые отдельной приглушённой секцией, не включая в сумму дня. Аналитика имеет глобальный тумблер «учитывать особые». Бюджет/структура считают всё как раньше.

---

## 1. Модель данных

Поле `special` на записи `expense`:

| Значение | Смысл |
|----------|-------|
| отсутствует / `null` | обычный расход |
| `'oneoff'` | разовое (крупная разовая покупка) |
| `'recurring'` | регулярное (аренда и т.п.) |

- Часть записи `expense` → синкается автоматически (last-write-wins по `updatedAt`), отдельной логики merge не нужно.
- `buildPayload()` **не меняем** — он режет только 5 top-level полей DB.
- Старые записи без поля = обычные. Обратная совместимость сохранена.
- При сохранении «Обычный» поле пишем как `null` (или не пишем), чтобы не засорять данные.

## 2. UI модалки расхода

Под полем комментария — сегмент из 3 чипов:

```
[ Обычный ]  [ Разовое ]  [ Регулярное ]
```

- По умолчанию выбран **Обычный**.
- Модульная переменная `_selectedExpSpecial` (паттерн как `_selectedGoalColor`), тонкий рендер-хелпер для перерисовки активного чипа.
- `openAddExpense()` / `editExpense()` ставят активный чип из поля записи (или «Обычный» для новой).
- `saveExpense()` пишет `special` (null если «Обычный»).

## 3. Day-таб (`renderDay`)

- `day-total` (большое число вверху) = сумма **только обычных** записей дня.
- Обычные записи → как сейчас: группировка по категориям, сортировка по сумме.
- Особые записи → отдельная секция внизу списка:
  - заголовок «Особые» + собственный подытог;
  - строки **приглушённые** (пониженная opacity / тусклый цвет);
  - НЕ входят в `day-total`;
  - маленькая метка типа в строке: разовое (напр. `•`) / регулярное (напр. `↻`);
  - свайп-удаление и правка ✎ работают как у обычных.
- Если особых нет — секция не рендерится.

Поиск / фильтр категорий (`renderExpenseSearch`) — показывает всё, включая особые, без специальной обработки (поиск идёт по всей базе).

## 4. Бюджет / структура

**Без изменений в подсчётах.** Особые по-прежнему учитываются в `getCatSpent()` и прогресс-барах (лимиты пользователь закладывает вручную, всё «в счёте»).

- В списке расходов категории (`renderCatExpensesList`) — лёгкая метка особого расхода (тот же значок разовое/регулярное), для консистентности. Подсчёты не трогаем.

## 5. Аналитика (`renderStats` + `renderSavingsRate`)

- Глобальный тумблер вверху Аналитики: **«Учитывать особые»**.
  - Состояние в `sessionStorage` под ключом `statsInclSpecial`.
  - По умолчанию **выкл** = особые вырезаны из всех графиков.
- Хелпер `_statsExps(exps)` → возвращает массив без особых, если тумблер выкл; иначе исходный.
- Прогнать через хелпер каждый сбор данных по расходам:
  - месячные тренды расходов (`renderStats` lastN);
  - разбивка по категориям / группам;
  - `renderSavingsRate` (lastN расходов).
- Доходы не трогаем.
- Переключение тумблера вызывает `renderStats()` (и перерисовку savings rate).

## 6. Кэш / версии

UI + CSS меняются (чипы, приглушённые строки, тумблер):
- бамп `?v=` у `css/app.css` в `index.html`;
- бамп `V` в `sw.js`.

## Вне scope (YAGNI)

- Отдельные лимиты для особых.
- Различение особых в самом бюджет-прогрессе.
- Фильтр особых в поиске.
- Изменения в Excel/JSON-экспорте (поле попадёт в JSON-дамп само).
